# Recon brief: player-watch

**Task:** Bring Flutter in-playback watch chrome to 1:1 with redesigned web (right-click party menu, top-right AV toggles, camera dock, unified auto-hide)

**Effort:** opus-xhigh  **DependsOn:** ['shell', 'design-tokens']

## Redesigned web (behavior source of truth)

The redesigned in-playback watch surface is composed by WatchView inside Party.tsx (app/client/src/pages/Party.tsx:195-546). WatchView owns a SINGLE auto-hide state `visible` (Party.tsx:217,246-250 `poke()` = setVisible(true) + 3000ms timer) that is threaded into BOTH the player chrome AND RoomControls so all chrome fades together. Mouse move re-pokes on desktop (Party.tsx:426 onMouseMove={poke}); phone uses tap/double-tap gestures (Party.tsx:305-335).

Player (app/client/src/components/Player.tsx) renders three visual layers over the video, all monochrome and boxless:
- TopBar (Player.tsx:854-897): a top-RIGHT row (justifyContent:'flex-end') of room-essential AV toggles ONLY — chat, mic, cam, hide-self, and a camera-LAYOUT toggle (float/dock). Flat `IconBtn` (Player.tsx:831-849): no box/border/fill; glyph is rgba(244,244,245,.62) at rest, brightens to #f4f4f5 on hover, danger=#e0655e (mic/cam off). Fades via `opacity: visible?1:0` (Player.tsx:860). NO back button, NO title, NO host/leave/party control here.
- DesktopControlBar (Player.tsx:1444-1518): one pinned-bottom row (play/pause, mono current-time, Scrubber, mono duration, VolumeControl, Settings, fullscreen) over THE ONE ALLOWED GRADIENT — a bottom-up black-alpha legibility scrim `linear-gradient(0deg, rgba(0,0,0,.8), transparent)` height 120 (Player.tsx:1478-1482). Stays visible while the settings menu is open even if idle-faded (Player.tsx:1470 `shown = visible || settingsOpen`).
- Scrubber (Player.tsx:1265-1385): thin monochrome bar, 4px→6px on hover/drag (Player.tsx:1295), track rgba(255,255,255,.14), buffered rgba(255,255,255,.28), played #f4f4f5; round thumb visible ONLY on hover/drag and ONLY when canControl (Player.tsx:1377-1382). Read-only for guests. Has trickplay-frame hover preview (Player.tsx:1358-1373, GET /api/library/items/:id/trickplay).

Right-click Watch Party menu lives OUTSIDE the player, in RoomControls (app/client/src/components/RoomControls.tsx:39-48): a window `contextmenu` listener — `if (event.shiftKey) return` (passes through the native browser menu), else preventDefault + setOpen(true) opening the host modal (participant roster, transfer-host, kick, collaborative toggle, sync-mode, copy link, back-to-lobby, end-party). Desktop shows NO persistent party pill; the only always-on RoomControls chrome is a top-LEFT red back button (RoomControls.tsx:116-119) plus toasts (top-center) and a host-only join-request sidebar. Design spec docs/watchparty-design/README.md:128-138 mandates exactly this: right-click opens the Watch Party menu, shift+right-click preserves the native menu, "Do not render a second desktop Watch Party pill over the top-right player controls," and chrome auto-hides after 3s.

Camera docking (Party.tsx): layoutMode 'float'|'dock' (from PartyContext). Float = CameraGrid (app/client/src/components/CameraGrid.tsx) draggable/resizable react-rnd PiP tiles over the video (Party.tsx:463). Dock = a fixed left column `Dock` (app/client/src/components/Dock.tsx, left:18 top:76 bottom:108 width:180) AND the video stage gets `marginLeft:210` to shrink (Party.tsx:447) with a .3s transition — the player is never remounted, only the surrounding box narrows. The layout toggle in TopBar flips the mode (Party.tsx:455). Chat = floating right-side panel Chat.tsx (glass, top:76 right:12 bottom:84 width min(300)) opened by a click-to-open right-edge tab (Party.tsx:479-490), the `c` key, or the chat icon.

Keyboard (SyncBridge, Player.tsx:645-692): space/k play-pause, arrows seek/volume, j/l ±10s, m mute, f fullscreen, c open chat — all authored through the sync engine (requestPlay/Pause/Seek). Push-to-talk = hold `T` (usePushToTalk.ts:54-61) momentarily opens the mic then returns to muted; no-op if already unmuted; releases on keyup/blur. hideSelf (useHideSelf.ts) is a local-only render drop of your own tile while still publishing; camera-off auto-hides self-view (Party.tsx:60). Sync socket events (hooks/useSyncPlay.ts): emits sync:hello/play/pause/seek/stall/report, listens sync:schedule/host_gone. A/V via useLiveKit.ts (GET /api/livekit/token?partyId=). Camera/mic toggles are wrapped in a guardToggle (Party.tsx:282-285, SyncBridge seekBridge) so device (re)acquisition never authors a spurious pause to the room.

## Current Flutter (divergence)

Flutter composes the watch surface in flutter_app/lib/app/screens/party_screen.dart (_ImmersiveParty, lines 280-452): a full-bleed Stack of [PlayerView | lobby] + FloatingCameraLayer + _PartyChrome (top) + chat slide-over. PlayerView (flutter_app/lib/player/player_view.dart) = Stack(VideoView, PlayerChrome) using a shared media_kit PlayerController (never remounted on layout changes — good).

What already works and is wired: playback/sync (onSeek→syncEngineProvider.requestSeek, party_screen.dart:355), LiveKit A/V (livekitProvider), chat (chatProvider/ChatPanel), host controls modal, hide-self, trickplay preview, subtitle side-loading, downloads/cached-span scrubber overlay, window-level fullscreen. `AppColors.accent` is #F4F4F5 near-white (flutter_app/lib/ui/tokens.dart:22) so the scrubber IS already monochrome — NOT a divergence.

Divergences from the redesigned web app + design guide:
1. NO right-click Watch Party menu. party_screen.dart wraps the stage only in MouseRegion(onHover)/Listener(onPointerDown) for poke (lines 367-370); there is no secondary-tap/onSecondaryTap handler. Host controls are reachable only via a persistent top-right icon (_PartyChrome, line 742-748) — the exact pattern docs/watchparty-design/README.md:130-132 replaces.
2. Persistent top-right acrylic cluster _PartyChrome (party_screen.dart:672-777, `_ChromeCluster` = sc.SurfaceCard blurred box, lines 780-803) holds mic/cam/hide-self/chat AND host-controls (manage_accounts) AND leave (close). This duplicates the Watch Party control over the top-right player area (regression per README.md:132,165) and is a glass BOX, whereas web uses flat, boxless monochrome IconBtn.
3. NO camera dock mode / layout toggle. Only FloatingCameraLayer (flutter_app/lib/ui/widgets/floating_camera_tile.dart) exists; camera_grid.dart has a `strip` layout but it is not mounted anywhere in the watch stage, and there is no float/dock toggle nor a video-shrink margin. Web has both modes + a TopBar layout toggle.
4. player_chrome.dart _TopBar (lines 746-785) renders a back button + title, and PlayerView is handed title=party.mediaItemId (RAW item id) + onBack=confirmLeave (party_screen.dart:348-349). Web Player has NO back/title in its top bar; this both duplicates the leave affordance (also in _PartyChrome close icon) and shows an ugly raw id.
5. Control-bar scrim is a FLAT solid opaque bar (_kChromeBar 0xD90A0A0B, player_chrome.dart:37,871) with a comment "no gradients per the design system" — but the redesigned web control bar explicitly uses the one allowed bottom-up black-alpha GRADIENT (Player.tsx:1478-1482).
6. TWO independent auto-hide timers: PlayerChrome._visible/idle 3s (player_chrome.dart:302-317) AND _ImmersiveParty._chromeVisible/idle 3s (party_screen.dart:320-327). They fade the player chrome and party chrome on separate timers/poke sources instead of the web's single `visible`.
7. NO push-to-talk. There is no hold-T mic behavior anywhere; only a mic toggle. player_chrome.dart _onKey (lines 527-561) has space/k, arrows, j/l, f, m but NO `c` (chat) and NO `t` (PTT) — both present on web.
8. AV toggle icon set/placement differs: web TopBar = chat/mic/cam/hide-self/LAYOUT; Flutter cluster = mic/cam/hide-self/chat/host/leave (no layout toggle).

## Gaps to close

- **Right-click Watch Party menu (desktop)** — Add an onSecondaryTapUp / Listener(kSecondaryButton) over the whole player stage in _ImmersiveParty (party_screen.dart) that opens the existing _HostControlsDialog (the Watch Party menu). This is the headline design-guide requirement (README.md:130). Flutter desktop has no native browser context menu, so shift+right-click has no native counterpart to preserve — right-click→menu is sufficient. Ensure the secondary-tap gesture sits above the video but does NOT consume primary taps used for poke/playback.  _(risk: Gesture layer over the media_kit video surface must not steal or reorder the primary-tap poke path or the keyboard focus that drives onSeek→requestSeek; wrong hit-test ordering could break wake-on-tap or transport authoring.)_
- **Duplicate Watch Party control over top-right** — Remove host-controls (manage_accounts) and leave (close) from the persistent top-right _ChromeCluster so the top-right carries ONLY room-essential AV toggles (README.md:132,165). Route host controls through the new right-click menu; move Leave/Back to a top-LEFT red back button (matching RoomControls.tsx:116-119). Keep the join-request notification + waiting badge reachable (web keeps the join-request sidebar always visible in RoomControls).  _(risk: Removing the only host-controls entry point without wiring the right-click menu first would strand approve/reject/transfer/end-party — sequence the changes so host actions are never unreachable.)_
- **Top-right AV cluster styling + layout toggle** — Restyle the top-right toggles from the acrylic sc.SurfaceCard box to flat, boxless monochrome icon buttons (rest rgba(244,244,245,.62) → hover #f4f4f5, danger #e0655e) matching web IconBtn (Player.tsx:831-849), and add a camera-LAYOUT (float/dock) toggle to the set to mirror web TopBar (Player.tsx:888-894).
- **Camera dock mode + video shrink** — Add a 'dock' layout: a fixed ~180px left column of camera tiles (reuse camera_grid.dart CameraGridLayout.strip / Dock.tsx equivalent) and shrink the player stage by an animated left margin (~210px, .3s) instead of overlaying floating tiles, toggled by the new layout button. Float mode keeps FloatingCameraLayer. CRITICAL: change only the layout constraints around PlayerView/VideoView — do NOT rebuild the PlayerController or the media_kit VideoView.  _(risk: Reparenting or re-keying VideoView/PlayerController when the margin changes would tear down and recreate the video texture and desync playback; the controller instance and VideoView State must be preserved across the layout switch.)_
- **Control-bar scrim: flat → gradient** — Replace the flat solid _kChromeBar/_kChromeScrim ColoredBox behind the transport bar (player_chrome.dart:37,871) with a bottom-up black-alpha LinearGradient (rgba(0,0,0,.8)→transparent) matching Player.tsx:1478-1482 — the one allowed legibility gradient.
- **Unified auto-hide** — Collapse the two independent 3s idle timers into one source of truth: lift a single `visible` into _ImmersiveParty and pass it down to PlayerChrome (replace its internal _visible/_idleTimer with a prop) so the player chrome, top-right AV toggles, and the top-left back button all fade together on the same 3s timer, staying visible while chat/settings menus are open (web: Player.tsx:1470, Party.tsx:246-250).  _(risk: PlayerChrome's idle timer also drives cursor hide and keyboard-focus wake; when externalizing visibility, keep the FocusNode/keyboard path intact so transport shortcuts and onSeek authoring still work.)_
- **Push-to-talk + c/t shortcuts** — Add hold-T push-to-talk (mirror usePushToTalk.ts: on keydown T open mic if currently muted, on keyup/blur return to muted, no-op if manually unmuted, guard key-repeat) and add `c` (toggle chat) to the keyboard handler. Wire through livekitProvider.setMic; independent of sync/playback.
- **Back-bar title cleanup** — Stop passing title=party.mediaItemId (raw id) to PlayerView and remove the back+title _TopBar from player_chrome.dart for the party path (web Player renders neither); rely on the top-left back button. If a title is ever shown, resolve the real media name, not the id.

- **Approach:** 1) Right-click menu: in _ImmersiveParty (party_screen.dart) wrap the stage in a Listener/GestureDetector handling the secondary (right) button; onSecondaryTapUp call _openHostControls(context) (the existing _HostControlsDialog = the Watch Party menu). Keep the primary-tap poke path and keyboard focus untouched; ensure the secondary handler is translucent to primary taps. 2) De-duplicate top-right: strip host-controls + leave from _ChromeCluster; keep only mic/cam/hide-self/chat and ADD a float/dock layout toggle. Restyle from sc.SurfaceCard box to flat boxless monochrome icon buttons (rest .62 white, hover full white, danger red). Add a top-LEFT red back button that fades with chrome. ## Open questions (resolved by plan decisions where applicable)
- Does the Flutter design system permit a bottom-up gradient scrim behind the transport bar? player_chrome.dart currently asserts 'no gradients per the design system', but the redesigned web control bar treats the black-alpha legibility gradient as the ONE allowed exception (README.md:31) — confirm the token/lint policy before switching from the flat bar.
- Where should host-only actions live once removed from the persistent top-right cluster? Web reaches them ONLY via right-click on desktop; confirm Flutter desktop should drop the always-visible host-controls icon entirely and rely on right-click, and whether a fallback (e.g. long-press) is wanted for trackpad users without a right button.
- Is a float/dock camera layoutMode expected to be persisted/shared via PartyContext like the web (party.layoutMode), or purely local UI state in Flutter? The web reads it from PartyContext; Flutter's party_provider may not carry it.
- Should the top bar show a real resolved media title (web shows none in-player)? Currently Flutter passes the raw mediaItemId as the title — confirm it should simply be dropped rather than resolved to a name.
