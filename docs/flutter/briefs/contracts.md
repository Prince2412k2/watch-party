## Current Flutter (divergence)

The Flutter transport layer is a generic passthrough, so it is largely already capable — the gaps are in the FROZEN CONSTANT reference and a couple of typed helpers, not the wire itself.

- flutter_app/lib/net/socket_client.dart: SocketClient is fully generic — emit(String,data), emitWithAck(String,data), on(String,handler). It carries the connect.sid cookie on the handshake via extraHeaders (socket_client.dart:66-75) and forces websocket transport + forceNew. Because it takes arbitrary event-name strings, it can technically emit/subscribe ANY event without code changes; the real risk is that higher layers reference the ClientEvent/ServerEvent constants which are incomplete.
- flutter_app/lib/net/events.dart: the frozen event vocabulary. ClientEvent (events.dart:60-83) has 22 constants; ServerEvent (events.dart:86-103) has 16. This file was transcribed from the OLD backend and its doc-comment header (events.dart:10-51) is stale.
- flutter_app/lib/data/api_client.dart: DioApiClient covers auth (login/me/logout), library (home/items/children/item/latest/search/trickplay/imageUrl), playback (nativeStreamUrl/playbackInfo), subtitles (content/upload/delete via the item-scoped routes), livekit (livekitToken), and servarr via generic servarrGet/servarrPost/servarrDelete (api_client.dart:401-433). The generic servarr passthrough covers every servarr path the redesigned web app hits (radarr/releases, radarr/grab, radarr/releases/cancel, sonarr/request-season, sonarr/language-profiles, downloads/enriched, downloads/:hash/detail, qbittorrent/*, manual/magnet, health, image).

Divergences from what the redesigned web client uses: (1) events.dart is MISSING three client emits the web app depends on — party:resume, browse:view, party:setPlaybackTracks — and one server event — chat:history. (2) servarrPost has no query-parameter support and no content-type control, so POST /api/servarr/manual/torrent (raw application/x-bittorrent body + ?service=&targetId=… query) cannot be issued cleanly. (3) servarrImageUrl builds `/api/servarr/image?url=<remoteUrl>` (api_client.dart:432-433) but the server contract expects `?service=radarr|sonarr&path=/MediaCover/…` (servarr/index.js:1193-1197) — a straight mismatch (low impact since most posters arrive as public remoteUrl and are loaded directly).

## Gaps to close

- **events.dart missing client emits** — Add ClientEvent constants party:resume ('party:resume'), browse:view ('browse:view'), and party:setPlaybackTracks ('party:setPlaybackTracks') to flutter_app/lib/net/events.dart. party:resume is how the web re-attaches to an in-progress party after a socket reconnect and receives chat:history + party:state (index.js:255-277); without it the Flutter client cannot resume a session on reconnect. browse:view carries the shared-screen tab/screen/mediaId/seasonId/episodeId navigation the redesigned web browse experience relies on (index.js:470-484) — plain browse:navigate only moves the drill stack. party:setPlaybackTracks lets the host change audio/subtitle tracks mid-playback and rebroadcast party:state (index.js:542-561).  _(risk: Watch-party sync/resume and shared-browsing parity break: missing party:resume means no reconnect recovery; missing browse:view means the shared-screen tab/detail navigation desyncs; missing setPlaybackTracks means host track changes never propagate to the room.)_
- **events.dart missing server event chat:history** — Add ServerEvent constant chatHistory ('chat:history') to flutter_app/lib/net/events.dart. The server pushes the full message backlog on party:resume, party:join and party:approve (index.js:274,325,386 via restoreSocket); the redesigned web listens for it (PartyContext.tsx:228).  _(risk: Chat history does not load on join/resume — the message pane starts empty instead of showing prior conversation.)_
- **events.dart stale doc header** — Update the doc-comment inventory (events.dart:10-51) so it lists party:resume, browse:view, party:setPlaybackTracks (client→server) and chat:history (server→client). It currently claims to be transcribed 'verbatim' from the backend but omits these, which will mislead the engineer building the party/sync layer.
- **servarrImageUrl contract mismatch** — Fix DioApiClient.servarrImageUrl (api_client.dart:432-433) to match GET /api/servarr/image?service=<radarr|sonarr>&path=</MediaCover/...> (servarr/index.js:1193-1197) instead of the non-existent ?url= param — or drop the helper if the client only ever renders the public remoteUrl posters directly.  _(risk: Any poster routed through the proxy 400s; low impact because most shaped responses carry a public remoteUrl loaded directly, but a MediaCover-only *arr image will fail to render.)_
- **servarrPost cannot do raw torrent upload** — Give servarrPost (api_client.dart:411-417) a way to send a raw application/x-bittorrent body plus query params, or add a dedicated method, so POST /api/servarr/manual/torrent?service=&targetId=&... (servarr/index.js:1111-1134) can be called. POST /api/servarr/manual/magnet already works via servarrPost with a JSON body.  _(risk: Manual .torrent-file submission (a downloads feature) is unreachable from Flutter; magnet submission still works, so partial functionality only.)_
