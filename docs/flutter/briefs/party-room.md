# Recon brief: party-room

**Task:** Rebuild Watch Party entry (bottom-right popcorn + expandable widget) and align in-party room/host controls with the redesigned web app

**Effort:** default  **DependsOn:** ['design-tokens', 'shell', 'chat', 'livekit', 'player-playback']

## Redesigned web (behavior source of truth)

The redesigned web app splits Watch Party into TWO surfaces:

(1) OUTSIDE playback — the shell popcorn control. `components/WebShell.tsx:270-277` renders a bottom-right floating cluster (`web-party-float`) with a circular popcorn button (`<img src="/popcorn.png">`, WebShell.tsx:273). Label toggles between "Watch party" (idle) and "N in party" (live, `session ? ...` WebShell.tsx:274) plus a red waiting-count badge `<b>{waitingCount}</b>` (WebShell.tsx:275). Clicking toggles `WebPartyWidget` (WebShell.tsx:111-161), a popover that: with NO session shows "Start a party" (createRoom, WebShell.tsx:127,244-256) + "Join with a code" (opens `JoinDialog`, WebShell.tsx:163-199 — an 8-char code input validated `/^[0-9A-F]{8}$/`, WebShell.tsx:174); with a session shows a QR (`PartyQr`, WebShell.tsx:149), room code + "Copy invite" (WebShell.tsx:150), the people roster with host + kickable guests (WebShell.tsx:152-155), a host-only "Waiting to join" list with approve/reject (WebShell.tsx:156), "Open player" → navigate(`/party/${id}`) (WebShell.tsx:157), and host "End party" with confirm (WebShell.tsx:158). Auto-opens for host when a guest is waiting (WebShell.tsx:226-228); closes on outside pointerdown (WebShell.tsx:235-242).

(2) INSIDE the `/party/:id` route — `pages/Party.tsx` + `components/RoomControls.tsx`. Party.tsx is a state machine keyed on `role` and `session.stage`: joinError → "Party not found" (Party.tsx:90-105); `role==='waiting'` → `Lobby.tsx` waiting-room (sonar pulse, party code, Leave, Lobby.tsx:1-91); `!session` → "Connecting…" spinner (Party.tsx:108-117); `session.stage==='lobby'` → embedded shared `Library` browse with a CodePill header, guest "host is choosing" banner, LobbyAVBar, RoomControls stage=lobby (Party.tsx:136-178); else WatchView (immersive player). `RoomControls.tsx` supplies the in-party chrome: toasts (RoomControls.tsx:90-97), a top-left Back arrow `leaveRoom()` that just does history.back/navigate('/library') WITHOUT ending the party (RoomControls.tsx:64-70,117-119), a host-only join-request sidebar with approve/reject that STAYS visible (RoomControls.tsx:137-155), and the Watch-party modal. Desktop opens the modal via right-click anywhere (`contextmenu` listener, Shift bypasses, RoomControls.tsx:39-48); phone gets a visible people-icon button with waiting badge (RoomControls.tsx:124-134). The modal (RoomControls.tsx:158-249): roster with HOST badge + per-guest transferHost/kick (host only, RoomControls.tsx:178-194), collaborative-control toggle "Let guests browse, play, pause & seek" (RoomControls.tsx:197-205), sync-mode Hopping/Dragging segmented control WITH a mode-dependent description (RoomControls.tsx:207-220), "← Pick something else" = backToLobby (watching+host, RoomControls.tsx:222-224), a QR + code + Copy-link share block (RoomControls.tsx:226-235, `JoinQR` inline SVG via qrcode pkg RoomControls.tsx:276-293), and a danger-zone "End party for everyone" with a separate confirm dialog (RoomControls.tsx:237-268). State + actions come from `context/PartyContext.tsx`: it holds session/role/toasts, emits create/createRoom/join/approve/reject/kick/transferHost/endParty/setCollaborative/setSyncMode/selectMedia/backToLobby (PartyContext.tsx:283-397) and drives toasts on party:waiting/user:joined/user:left/party:rejected/party:kicked/party:ended/host:changed (PartyContext.tsx:143-203). Socket contract per net/events.dart: party:create/join/approve/reject/kick/end/transferHost/setCollaborative/setSyncMode/selectMedia/backToLobby (client→server) and party:state/waiting/approved/rejected/kicked/ended, host:changed, user:joined/left, camera:removed (server→client).

## Current Flutter (divergence)

Flutter has NO popcorn control and NO expandable shell widget. The only party affordance outside the route is `ui/widgets/global_party_bar.dart` — a slim 40px horizontal strip mounted app-wide in `app/app.dart:67` ABOVE routed content. It shows only when a party exists and hides on `/party*` (global_party_bar.dart:33-37); renders "status · N in party · id" + "Return to party" + "End"/"Leave" buttons (global_party_bar.dart:44-88). There is no create/join entry point in the shell at all — entry is only by navigating to the `/party` route. The shell itself (`app/screens/app_shell.dart:100-117`) is still the OLD design: a LEFT NavRail + top-right profile + a top `_WindowBar` with a section title (`app/app.dart`), i.e. exactly the nav-rail/top-bar layout the redesign abandons; there is no bottom-centered nav and no bottom-right popcorn slot.

The in-party route is `app/screens/party_screen.dart`. Pre-join `_PartyLobby` (party_screen.dart:133-270) is a full-screen card with "Start a party" (create, :220-226) + a plain "Party ID" text field (no 8-char/hex validation, no QR scan) + "Join party" (:242-254); the waiting state is a CircularProgressIndicator + "Waiting to be let in" text (:152-192), not the web's sonar waiting-room. In-party `_ImmersiveParty` (party_screen.dart:280-452) is a full-bleed Stack: PlayerView/lobby stage, FloatingCameraLayer, `_PartyChrome`, chat slide-over. `_PartyChrome` (:672-777) is a PERSISTENT top-right acrylic cluster of icon buttons — mic, cam, hide-self, chat, host-controls (host only, `Icons.manage_accounts_outlined` with waiting badge, :741-748), and a red Close (:749-755) whose tooltip is "End party"/"Leave" and which calls `_confirmLeave` → actually ends/leaves the party (:431-451). Room-code pill is top-left (:934-989); host-only join-request card sits under the cluster (:993-1086). `_HostControlsDialog` (:1173-1385) is the modal: roster with star+HOST badge, make-host, kick, context menu (:1390-1475); collaborative Switch labeled "Let guests play, pause & seek" (missing "browse", :1255-1279); sync-mode toggle-group WITHOUT the mode description line (:1481-1501); watching-only "Switch movie" (a private media picker sheet, :1290-1301) + "← Back to lobby" (:1303-1311); a "Share this code" block with Copy but NO QR (:1314-1344); danger-zone "End party for everyone" with confirm (:1346-1366). `state/party_provider.dart` maps the session and correctly separates backToLobby()="Stop Movie" (:339-340) from end()="Stop Stream" (:342-346), derives isHost/canControl (:33-35), and subscribes to partyState/waiting/approved/rejected/kicked/ended, host:changed, user:joined/left (:74-121) — but emits NO toasts on any of those events (only copy-code toasts, :1505-1538), unlike PartyContext. ## Gaps to close

- **Popcorn control + expandable widget (outside playback)** — Add a bottom-right circular popcorn control (reuse app/client/public/popcorn.png as a bundled Flutter asset) in the shell, replacing/augmenting GlobalPartyBar. Idle → label 'Watch party'; live → 'N in party' with a red waiting-count badge; host auto-opens it when someone is waiting. Clicking opens an expandable widget mirroring WebPartyWidget: no-session = Start a party (create) + Join with a code (validated 8-char hex dialog); live = QR + room code + Copy invite + roster (kick guests) + host approve/reject waiting list + 'Open player' (go /party/:id) + host 'End party'. This is the #1 missing surface and is an explicit design-doc requirement (README §Watch Party, checklist 'Popcorn control remains bottom-right outside playback').  _(risk: Popcorn must read the same app-wide, non-autoDispose partyProvider so it survives navigation; wiring a duplicate provider would desync roster/waiting from the party route.)_
- **QR invitation** — Add a client-side QR (e.g. qr_flutter package) rendering the join URL `${origin}/party/${id}` in BOTH the popcorn widget and the _HostControlsDialog 'Share this code' block. Web renders QR in both WebPartyWidget (PartyQr) and RoomControls (JoinQR); Flutter has neither. QR is an explicit design-doc invariant ('QR invitation').  _(risk: Must build the join URL from the real server origin the client is pointed at (AppConfig), not a hardcoded host, or scanned links break.)_
- **Party activity toasts** — Wire PartyNotifier socket handlers to emit toasts matching PartyContext: 'X wants to join' (warning) on party:waiting, 'X joined'/'X left' on user:joined/left, 'You are now the host' (success) when host:changed matches me, 'The host ended the party' on party:ended, 'The host declined your request' on party:rejected, 'You were removed' on party:kicked. Reuse the existing _showPartyToast/sc.showToast helper. Flutter currently shows zero party toasts.  _(risk: Toasts fire from provider socket handlers that run app-wide; ensure they don't double-fire on reconnect resync of party:state.)_
- **Back button = minimize, not end/leave** — Change the party route's Back/Close affordance to MINIMIZE (navigate to the shell, keep the party alive so the popcorn shows 'N in party'), matching web's leaveRoom() which is just history.back. End/leave should happen only from the host-controls danger zone (host) or an explicit leave in the popcorn/roster. Currently party_screen _confirmLeave (:431-451) and the red chrome Close (:749-755) always end/leave.  _(risk: If left as-is, the persistent-party + popcorn-reentry model is unreachable: any back gesture tears down socket/LiveKit/sync. Must ensure minimize does NOT call _leaveLocal (which disconnects the socket, party_provider.dart:358-375).)_
- **Desktop right-click to open Watch Party menu** — On desktop, open the host-controls modal via a secondary-tap/right-click GestureDetector over the player (Shift+right-click should pass through), and drop the persistent top-right host-controls icon button so desktop has no standalone Watch-party pill over the player controls. Keep a visible button only as the touch affordance. Web: RoomControls.tsx:39-48 (contextmenu) desktop, RoomControls.tsx:124-134 visible button phone; design-doc rule 'Do not render a second desktop Watch Party pill'.  _(risk: canControl/isHost gating must still decide what the menu shows; a right-click handler must not swallow the player's own transport interactions.)_
- **Lobby waiting-room visual** — Replace the plain spinner+text waiting state (_PartyLobby waiting branch, party_screen.dart:152-192) with the redesigned sonar-pulse waiting room from Lobby.tsx: 'WAITING ROOM' eyebrow, 'The host hasn't started yet' heading, party-code focal element, quiet 'Leave party' text button.
- **Collaborative-control label + sync-mode description** — Update the collaborative toggle copy to 'Let guests browse, play, pause & seek' (Flutter drops 'browse', :1265) and add the mode-dependent description line under Sync mode ('Everyone waits for the slowest viewer' for dragging / 'Host never waits; slow viewers catch up' for hopping), matching RoomControls.tsx:200,210-212.
- **Lobby stage = shared collaborative browse (cross-surface)** — Web lobby renders the full embedded Library so everyone browses together (Party.tsx:136-178) with a CodePill header and guest 'is choosing…' banner + mirror cursor; Flutter _LobbyStage is a static card + private picker. Aligning the lobby chrome (CodePill/choosing-banner) belongs here; the shared-browse/mirror mechanics depend on the library/shell surface and browse:navigate/browse:pointer events not yet implemented in Flutter.  _(risk: Full parity requires wiring browse:navigate/browse:state/browse:pointer and gating drive on canControl; out of scope for party-room alone but blocks true lobby parity.)_

## Implementation brief

- **Approach:** 1) Add a qr package (qr_flutter) and the popcorn.png asset to pubspec. 2) Build PopcornControl: a bottom-right circular float in AppShell (place in the AppShell Stack, bottom-right, mirroring the top-right profile button; longer term the shell moves to bottom-centered nav, but for this task just add the bottom-right popcorn). It watches partyProvider: idle shows popcorn+'Watch party'; live shows 'N in party' + red waiting badge; host auto-expands when partyWaitingProvider is non-empty. 3) Build PartyWidget popover: no-session → Start a party (partyProvider.notifier.create()) + Join with a code (JoinCodeDialog, validate ^[0-9A-F]{8}$ then notifier.join, route to /party/:id on joined); live → PartyQr(joinUrl), room code + Copy, roster with host + kickable guests, host approve/reject waiting list, 'Open player' (context.go /party/:id), host 'End party' (confirm → notifier.end()). Build joinUrl from AppConfig origin. 4) Repurpose GlobalPartyBar: either remove it in favor of the popcorn's live state, or keep it minimal; the popcorn is now the primary re-entry. 5) In party_screen: change the Back/Close affordance to MINIMIZE (context.go('/home') WITHOUT notifier.end()/leave() — party stays alive); move host-controls opening on desktop to a right-click (onSecondaryTap / Listener for kSecondaryButton, Shift passthrough) over the player and drop the persistent desktop host-controls pill, keeping a visible touch button; swap the waiting branch for the sonar waiting-room; add QR to _HostControlsDialog share block; fix collaborative label to include 'browse'; add the sync-mode description line. 6) In party_provider: emit toasts (reuse _showPartyToast/sc.showToast pattern) from the partyWaiting/userJoined/userLeft/hostChanged/partyRejected/partyKicked/partyEnded handlers, matching PartyContext copy. Keep backToLobby()=Stop Movie and end()=Stop Stream exactly as-is; do NOT conflate them.
